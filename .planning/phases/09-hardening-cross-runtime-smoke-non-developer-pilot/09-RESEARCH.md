## Summary

Phase 9 is the **v1 launch hardening phase** — no new feature primitives, only verification + pruning + documentation. The substrate from Phases 1-8 is already complete: runtime detection (FND-06), text_mode fallback (FND-06), surface cap source (Phase 2 D-09), state.brief.* allowlist (Phase 2 D-21), atomic commit infrastructure (Phase 1), and the four canonical gates (ALIGN/AUDIENCE/COMPLIANCE/CC-03). Phase 9 only **closes** what is already in flight: (1) HRD-01 verifies multi-runtime parity by stub-driven mock subprocess, no real Codex/Gemini/OpenCode CLI calls; (2) HRD-02 prunes the 68-command inherited GSD surface to the locked 12-command lineup with backup-branch preservation; (3) HRD-03 ships the rich `/brief-help` command (currently a thin pass-through wrapper to a hardcoded reference) with 4D categorization + partial-keyword match + inline Levenshtein top-3 suggestions; (4) HRD-04 logs the single dogfooding-session friction journal with explicit "remaining 2/3 deferred to v1.1" Out of Scope acceptance; (5) HRD-05 (added 2026-04-18) closes Phase 1 HALT-ACCEPTED residuals (a)+(b) only — 19 missing-file tests + 14 ARCHITECTURE.md count drift — leaving (c)+(d) for v1.1 with the empirical target ≤16 npm test failures (EMPIRICAL_BASELINE 6 + DELTA_CAP 10 inherited).

The two highest-stakes risks are (a) **HRD-02 surface deletion regressions** — `bin/install.js` registers commands at multiple anchor points; deleting 50+ command markdown files without parallel `bin/install.js` cleanup leaves dangling registrations; mitigated by Plan 02 enumerating both deletions in a single atomic commit. (b) **HRD-05(a) per-test triage drift** — for each of the 19 missing-file failures, the executor must judge "create file with placeholder content (workflow is real)" vs "delete the assertion (workflow is intentionally absent in v1)". Wrong calls re-create the dual-vocabulary problem Phase 1 was designed to eliminate. Mitigated by encoding the locked 12-command lineup (A-D01) as the explicit triage rubric: workflows referenced by the 12 locked commands → create; workflows referenced only by deleted commands → delete assertion.

**Primary recommendation:** Execute Phase 9 as 6-8 plans across 2-3 waves with **Wave 1 parallelism cap** (Phase 8 cwd-bug lesson — 3 worktree agents leaked into main during Phase 8 Wave 1; subsequent waves run single-agent in worktree-isolation). Treat HRD-02 surface pruning as the architectural keystone — deletions touch 50+ command files + `bin/install.js` cleanup + backup-branch preservation discipline + SURFACE-AUDIT.md generation in one atomic commit. The other four requirements (HRD-01/03/04/05) are independent and parallelizable in Wave 1. Do NOT add any new gate vocabulary, NOT add any runtime dependency, NOT add any new agents.

**Why no `fast-levenshtein` / `levenshtein` package:** A1 zero-runtime-deps rule. Iterative two-row DP is ~30 LOC and sufficient for typical CLI input lengths (≤25 chars). Performance is irrelevant at single-input invocation scale. Adding a runtime dep just for Levenshtein would break the lightness promise that lets BRIEF ride into Codex/Gemini/OpenCode without friction.

**Why no `commander` for help command:** A1 zero-runtime-deps rule. The /brief-help "command" is a markdown file (`commands/brief/help.md`) that the runtime parses; the underlying invocation goes through `brief-tools.cjs case 'help'` dispatcher. No CLI argument parser library needed.

**Why no `cross-spawn` / `execa` for HRD-01:** Built-in `node:child_process.spawn` works on macOS/Linux/Windows for our use case (spawning Node child processes with custom env). Cross-platform shell-quoting concerns don't apply because we never invoke through a shell; we invoke `process.execPath` directly with arg array.

### Pattern 2: Inline Two-Row Levenshtein DP (Zero-Dep)

**What:** Iterative dynamic-programming Levenshtein distance with O(min(m,n)) space and O(mn) time, ~30 LOC.

**When to use:** HRD-03 typo suggestion (C-D03). NEVER add `fast-levenshtein` runtime dep (A1 violation).

**Example:**
```javascript
// Source: canonical two-row DP, e.g., en.wikipedia.org/wiki/Levenshtein_distance#Iterative_with_two_matrix_rows
// A1-compatible (zero deps); ~30 LOC; sufficient for ~25-char command names

function levenshtein(a, b) {
  if (a === b) return 0;
  if (a.length === 0) return b.length;
  if (b.length === 0) return a.length;
  // Ensure a is the shorter — minimizes inner-loop allocations
  if (a.length > b.length) { const t = a; a = b; b = t; }

  let prev = new Array(a.length + 1);
  let curr = new Array(a.length + 1);
  for (let i = 0; i <= a.length; i++) prev[i] = i;

  for (let j = 1; j <= b.length; j++) {
    curr[0] = j;
    for (let i = 1; i <= a.length; i++) {
      const cost = a[i - 1] === b[j - 1] ? 0 : 1;
      curr[i] = Math.min(
        prev[i] + 1,        // deletion
        curr[i - 1] + 1,    // insertion
        prev[i - 1] + cost  // substitution
      );
    }
    [prev, curr] = [curr, prev];
  }
  return prev[a.length];
}

// Usage in /brief-help typo handler:
function suggestTopK(input, candidates, k = 3, maxDistance = 3) {
  return candidates
    .map(c => ({ name: c, distance: levenshtein(input, c) }))
    .filter(r => r.distance <= maxDistance)
    .sort((x, y) => x.distance - y.distance)
    .slice(0, k);
}
```

### Pattern 3: Static JSON Catalog with Runtime Cache

**What:** On first `/brief-help` invocation, scan `commands/brief/*.md` frontmatter, build JSON catalog `{ name, description, category, body }`, cache in module-level variable.

**When to use:** HRD-03 categorized listing (C-D01). Cache survives within a single Node process; recomputed on next process.

**Example:**
```javascript
// brief/bin/lib/help.cjs (NEW Phase 9)
const fs = require('node:fs');
const path = require('node:path');
const { extractFrontmatter } = require('./frontmatter.cjs');

const PHASE_CATEGORIES = {
  define: 'DEFINE', discover: 'DISCOVER', design: 'DESIGN',
  deliver: 'DELIVER', export: 'DELIVER', 'add-workstream': 'DESIGN',
  status: 'HELPERS', help: 'HELPERS', 'init': 'HELPERS',
  'progress': 'HELPERS', 'undo': 'HELPERS', 'pause-work': 'HELPERS',
};

let _catalogCache = null;

function buildCatalog(commandsDir) {
  if (_catalogCache) return _catalogCache;
  const files = fs.readdirSync(commandsDir).filter(f => f.endsWith('.md'));
  const entries = files.map(f => {
    const slug = f.replace(/\.md$/, '');
    const content = fs.readFileSync(path.join(commandsDir, f), 'utf-8');
    const fm = extractFrontmatter(content) || {};
    return {
      slug,
      name: fm.name || `brief:${slug}`,
      description: fm.description || '',
      category: PHASE_CATEGORIES[slug] || 'HELPERS',
      body: content.replace(/^---[\s\S]*?---\n/, ''), // body without frontmatter
    };
  });
  _catalogCache = entries;
  return entries;
}

function renderCategorized(catalog) {
  const groups = { DEFINE: [], DISCOVER: [], DESIGN: [], DELIVER: [], HELPERS: [] };
  for (const e of catalog) groups[e.category].push(e);
  let out = '# BRIEF Command Reference\n\n';
  for (const cat of ['DEFINE', 'DISCOVER', 'DESIGN', 'DELIVER', 'HELPERS']) {
    out += `## ${cat}\n\n`;
    for (const e of groups[cat]) out += `- \`/brief-${e.slug}\` — ${e.description}\n`;
    out += '\n';
  }
  return out;
}

module.exports = { buildCatalog, renderCategorized, /* ... levenshtein, suggestTopK ... */ };
```

## Common Pitfalls

### Pitfall 1: HRD-02 Surface Pruning Leaves Dangling `bin/install.js` Tuples

**What goes wrong:** `commands/brief/{deleted}.md` is removed but `bin/install.js` still has SRC tuples or path references to that command, causing install errors or stale entries in user config dirs.

**Why it happens:** Phase 1 Plan 08 already established that `bin/install.js` has 100+ SRC/PATCHES_DIR_NAME/MANIFEST_NAME/cache path/$gsd-* references. The cleanup pattern requires touching multiple anchor points per deletion.

**How to avoid:**
1. Plan 02 task: enumerate every `bin/install.js` reference for each of the 56 deletions.
2. Run `grep -n "{cmd-name}" bin/install.js` for each deleted command before committing.
3. Verify post-pruning: `node bin/install.js --help` succeeds, `npm test` for installer tests pass.

**Warning signs:** `tests/agent-install-validation.test.cjs` or `tests/brief-discover-parallel-smoke.test.cjs` failing with "command not found" or "duplicate registration" after pruning commit.

### Pitfall 2: HRD-05(a) Triage Mis-Calls "Create" When File Should Be Deleted

**What goes wrong:** Executor sees a missing-file test and reflexively creates a placeholder file, even though the referenced workflow is one of the 56 deleted commands and SHOULD have its assertion deleted.

**Why it happens:** The 19 missing-file tests reference 3 files (`brief/workflows/pr-branch.md`, `brief/workflows/diagnose-issues.md`, `brief/references/ui-brand.md`). Without checking which file ties to which locked-12-command, the executor defaults to the easier "create placeholder" path.

**How to avoid:**
- Plan 05 task includes per-file rubric:
  - `pr-branch.md`: tied to milestone PR workflow — **does any of locked 12 commands invoke it?** If no → delete assertion. (Likely: delete; PR-branch flow was inherited GSD developer surface.)
  - `diagnose-issues.md`: tied to `gsd-debugger` agent which is intentionally absent post-Phase-1 (FND-02 dev-surface removal) — **delete assertion** (referenced workflow is permanently gone in v1).
  - `ui-brand.md`: tied to `gsd-ui-researcher`/`gsd-ui-checker` agents intentionally absent post-Phase-1 — **delete assertion**.
- Per-test single-line rationale logged (e.g., `tests/bug-2004-pr-branch-milestone.test.cjs: assertion deleted because pr-branch.md is GSD-only and not in locked 12-cmd lineup`).

**Warning signs:** Post-closure npm test count exceeds 16 because executor created placeholder files that then trigger downstream "file is empty" assertions in other tests.

### Pitfall 3: HRD-03 Levenshtein False Positives on Short Commands

**What goes wrong:** With distance ≤ 3 threshold and command names as short as 4 characters (`undo`, `help`), Levenshtein suggests irrelevant matches (e.g., `undo` ↔ `redo` would have distance 2 — but BRIEF has no `redo`; `help` ↔ `held` distance 1).

**Why it happens:** Levenshtein is purely character-level and has no semantic awareness; short strings are dominated by random small distances.

**How to avoid:**
- Within the 12-command lineup, manually verify the cross-distance matrix of all 12 names. If two locked commands have distance ≤ 3 between them, the typo handler may suggest the wrong one.
- Quick check (planner runs once during plan-phase):
  ```bash
  # All 12 command names (slugs):
  # define, discover, design, deliver, export, add-workstream,
  # status, help, init, progress, undo, pause-work
  ```
  Empirically: most distances are well above 3 (e.g., `define` ↔ `deliver` = 4; `define` ↔ `discover` = 4; `define` ↔ `design` = 2 — close!). The `define`/`design` collision is real and acceptable: a user typing `desin` could mean either; suggesting both with distance 2 is correct.
- Document the known false-positive: distance ≤ 3 may suggest both `define` and `design` for typos near either; this is acceptable UX.

**Warning signs:** `tests/brief-help-levenshtein.test.cjs` flakiness on edge-case inputs; pilot user reports "weird suggestion."

### Pitfall 4: HRD-01 Mock Subprocess Hangs on Stalled AskUserQuestion

**What goes wrong:** Mock subprocess invokes a `brief-tools.cjs` command that hits an `AskUserQuestion` call but `text_mode` flag wasn't propagated correctly, causing the child process to wait for user input forever.

**Why it happens:** The runtime detection chain has multiple layers (env → config → core defaults); any one being out of sync breaks the fallback.

**How to avoid:**
- Mock subprocess MUST set both `INSTRUCTION_FILE` env AND pass `--text` CLI flag (belt + suspenders).
- HRD-01 stub uses `execFileSync` with `timeout: 5000` (5s) — any cell taking > 5s is auto-marked FAIL with timeout reason.
- Each mock cell records both stdout AND exit code; non-zero exit + AskUserQuestion in stdout = FAIL with reason "text_mode fallback not honored".

**Warning signs:** SMOKE-TEST.md cell shows FAIL with reason "timeout 5s" — indicates the runtime config plumbing has a bug introduced after Phase 1 FND-06 verification.

### Pitfall 5: HRD-04 Pilot Journal Becomes Performative

**What goes wrong:** The single 1-of-3 pilot journal becomes a rationalization document ("here's why each of these frictions is fine actually") rather than a faithful record.

**Why it happens:** The pilot is the same person doing the dogfooding (Korean non-technical product owner = the user driving BRIEF on BRIEF). Pitfall #14 dogfooding trap.
